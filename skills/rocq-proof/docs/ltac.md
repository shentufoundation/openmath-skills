# Proof Automation

### 1. Ltac: The Traditional Tactic Language

Ltac allows you to build complex tactics by combining simpler ones with control flow, loops, and pattern matching.

**Control Flow and Branching**

- **Sequence (`;`)**: `tac1 ; tac2` applies `tac1` to the current goals, and then applies `tac2` to all subgoals generated by the first tactic.
- **Backtracking Branching (`+`)**: `tac1 + tac2` tries `tac1` first. If `tac1` fails, or if a subsequent tactic later fails, Ltac backtracks and tries `tac2`.
- **First Progress (`||`)**: `tac1 || tac2` applies `tac1`, but if it fails to make progress, it falls back to `tac2`.
- **First Success (`first`)**: `first [ tac1 | tac2 ]` applies the first tactic that succeeds, but does not allow backtracking into the alternatives if a later tactic fails.
- **Loops**: `repeat tac` applies `tac` recursively until it fails or stops making progress. `do N tac` repeats the tactic exactly `N` times.
- **Error Handling**: `try tac` catches errors from `tac` and leaves the goal unchanged instead of failing.

**Pattern Matching**

- **Term Matching**: `match term with | pattern => tac end` sequentially matches a term against patterns and substitutes the found metavariables into the tactic.
- **Goal Matching**: `match goal with | H : pattern |- conclusion_pattern => tac end` examines the local context and conclusion. It supports non-linear matching (using the same metavariable multiple times to enforce equality).
- **Context Patterns**: `context [ cpattern ]` matches any term that contains a subterm matching the `cpattern`.

**Success and Failure**

- **`fail` / `gfail`**: `fail` aborts the current tactic and triggers backtracking. `gfail` fails globally even when there are no goals left.
- **`assert_succeeds` / `assert_fails`**: Tests whether a tactic succeeds or fails without actually modifying the proof state.

### 2. Ltac2: The Modern, Typed Tactic Language

Ltac2 is an ML-family, call-by-value functional language featuring Hindley-Milner static typing. It aims to resolve Ltac1's unpredictable semantics and performance issues.

**Core Features**

- **Types**: Built-in types include `int`, `string`, `exn` (exceptions), `constr` (kernel-side terms), and `pattern`. You can also define custom algebraic datatypes and records.
- **First-Class Backtracking**: Backtracking in Ltac2 is modeled as streams of values, controlled explicitly via the `Control` module. `Control.plus` appends two backtracking streams, while `Control.zero` initiates a failure.
- **Goal Access**: Goals are accessed explicitly via primitives like `Control.hyp` and `Control.goal`, rather than being implicitly available everywhere.

**Meta-Programming and Antiquotations**

- **Quotations**: Ltac2 makes term parsing explicit. You construct terms using `constr:(...)` for closed terms or `open_constr:(...)` for terms with holes.
- **Antiquotations**: You can insert Ltac2 variables into Rocq terms using the `$x` syntax, which is equivalent to `ltac2:(Control.refine (fun () => x))`.

**Pattern Matching on Terms and Goals**

- **`lazy_match!`**: Commits to the first matching branch and does not backtrack if the right-hand tactic fails.
- **`match!`**: Tries the next branch if the right-hand tactic fails, but prevents subsequent tactics outside the match from triggering backtracking into the branches.
- **`multi_match!`**: Allows full backtracking; failures occurring after the match will trigger the selection of the next matching branch.

**Interoperability with Ltac1**

- You can call Ltac1 from Ltac2 using the `ltac1:(...)` quotation, which evaluates for side-effects and returns `unit`.
- You can call Ltac2 from Ltac1 using the `ltac2:(...)` quotation.

### 3. SSReflect: Small-Scale Reflection

SSReflect provides an alternative, highly structured proof language designed to keep scripts maintainable by managing the context explicitly.

**Bookkeeping**

- **Discharge (`:`)**: Moves facts and constants from the context into the goal's conclusion (generalization). For example, `move: H` pushes hypothesis `H` into the goal.
- **Introduction (`=>`)**: Moves variables from the goal's conclusion into the local context. For example, `move=> x H` introduces variable `x` and hypothesis `H`.

**Views and Intro Patterns**

- **Simplification**: Using `=> //` solves trivial subgoals with the `done` tactic, while `=> /=` performs partial evaluation (simplification).
- **Views (`/`)**: Applies a lemma to a hypothesis on the fly. For example, `move=> /lemma1 H` applies `lemma1` to the top assumption before naming it `H`.
- **Branching (`[ | ]`)**: Destructs inductive types. For instance, `case=> [H1 | H2]` performs case analysis and names the resulting hypotheses in the two subgoals.

**Forward Reasoning**

- **`have`**: Starts a subproof for an intermediate result. `have H : P` asks you to prove `P`, then makes it available as `H`.
- **`suff`**: The inverse of `have`; it asserts that an intermediate step `L` implies the main goal, asking you to prove `L -> Goal` first, and then `L`.
- **`wlog` (without loss of generality)**: Introduces a cut to reduce a problem to a more specific case, often used with symmetry arguments.

**Advanced Rewriting**

- **Chained Rewrites**: `rewrite !lemma1 -lemma2 /def //=` rewrites `lemma1` as many times as possible (`!`), rewrites `lemma2` right-to-left (`-`), unfolds `def` (`/`), and then simplifies and solves trivial goals (`//=`).
- **Contextual Patterns**: `rewrite [in X in _ = X]rule` precisely targets where a rewrite should occur by specifying the surrounding context.
- **Under Binders**: The `under` and `over` tactics allow for rewriting inside lambda abstractions using extensionality lemmas.
